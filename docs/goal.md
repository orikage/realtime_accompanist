# /goal

## Project Goal

このプロジェクトでは、MIDIキーボードで右手メロディを弾くと、直近フレーズからキー・コード候補を推定し、ドラム・ベース・コード伴奏をリアルタイム生成する展示向けプロダクトを開発する。

目的は、Maker Faireやハッカソンで実演できる「右手だけでバンドがついてくるAI伴奏楽器」のMVPを作ることである。

プロジェクト資料として、事前に作成された `realtime_accompanist_codex.zip` を参照する。  
ZIP内の要件、ライブデモ例、実現可能性、アーキテクチャ、MVP計画、タスク分解を初期コンテキストとして扱う。

ただし、ZIPの内容は絶対仕様ではない。  
実装中により良い判断があれば更新してよいが、その場合は理由・影響範囲・代替案を記録する。

---

## Primary Objective

最優先は、展示で安定して動くMVPを作ることである。

最初に実現する体験は以下。

> 右手メロディを弾くと、キーとコード候補がリアルタイムに動き、簡単な伴奏が破綻せず鳴る。

この体験を最短で成立させる。

---

## Initial Planning Policy

初期叩き台、初期設計、最初の開発プランは Claude Code に担当させる。

理由は、最初のプランがプロジェクト全体の方向性、設計品質、スコープ、品質担保方針を決めるためである。  
この段階ではコストよりも、設計の妥当性、スコープ制御、リスク洗い出し、レビュー観点の品質を優先する。

Claude Code には最初に以下を作成させる。

- プロジェクト全体方針
- MVPスコープ
- 非ゴール
- 技術選定
- リポジトリ構成案
- Milestone 1〜3
- 最初のタスク分解
- 品質ゲート
- テスト方針
- Codex / Gemini / Claude の役割分担
- リスク一覧
- 展示デモに向けた優先順位

Claude Code の初期プラン作成が完了するまで、コア実装には入らない。

## Agent Assignment Policy

AI CLIは、コストと得意分野で使い分ける。

---

### Claude Code

Claude Code は、初期設計、開発計画、仕様整理、設計レビュー、品質レビューを担当する。

特に最初の叩き台は Claude Code に作らせる。

Claude Code に任せる作業:

- 初期開発プラン作成
- 要件整理
- MVPスコープ定義
- 非ゴール定義
- アーキテクチャ設計
- タスク分解
- 品質基準作成
- テスト方針作成
- リスク洗い出し
- 実装方針レビュー
- 変更差分レビュー
- READMEやデモ台本の改善
- 複雑な設計判断

Claude Code は、直接実装よりも、プロジェクトの方向性と品質ゲートを担う役割を重視する。

---

### Codex

Codex は、主実装、コード変更、統合、リファクタリングを担当する。

Codex に任せる作業:

- リポジトリ初期化
- Claude Code の初期プランを元にした実装
- MIDI入力処理
- メロディバッファ
- キー推定
- コード推定
- 伴奏エンジン
- UI連携
- Windows起動スクリプト
- テスト修正
- バグ修正
- リファクタリング
- Gemini が作った初稿の統合と品質改善

Codex はコードベースの整合性を保ちながら、実装を前に進める中心担当とする。

---

### Gemini CLI

Gemini CLI は安価に回せるため、簡単な実装、単純作業、初稿作成を優先的に担当する。

Gemini CLI に任せる作業:

- 小さな関数実装の初稿
- テストケースの初稿
- JSON/YAML/Markdown作成
- サンプルデータ作成
- 伴奏パターンJSON作成
- READMEの初稿
- CLIスクリプトの初稿
- 単純なリファクタリング
- 型定義や設定ファイル整備
- TODO洗い出し
- ドキュメント整形

ただし、Gemini CLI の出力は必ずレビュー対象とする。  
安価なため積極的に使うが、品質保証なしにmainへ入れない。

## Quality Assurance Policy

このプロジェクトでは、AIに実装を任せるが、品質担保を非常に重視する。

---

### Q1. 実装は小さなタスクに分割する

一度に大きな機能を実装しない。

悪い例:

```text
MIDI入力、キー推定、コード推定、伴奏生成、UIを全部作って
```

良い例:

```text
まずMIDI入力イベントを受け取り、note_on/note_offをログ表示するだけの実装を作る
```

---

### Q2. 変更前に実装計画を出す

各タスクでは、実装前に以下を出す。

```text
- 変更対象ファイル
- 実装方針
- 追加する関数/クラス
- 想定されるリスク
- テスト方法
```

Gemini CLI であっても、簡単な実装であっても、最低限の実装計画を出す。

---

### Q3. 変更後にセルフレビューを出す

実装後、担当AIは以下を出す。

```text
- 何を変更したか
- なぜその実装にしたか
- 動作確認方法
- テスト結果
- 未解決の懸念
- 追加すべきテスト
```

---

### Q4. コア部分は必ず別AIレビューを通す

以下の領域は、実装者とは別のAIにレビューさせる。

* キー推定
* コード推定
* 伴奏生成
* MIDI入出力
* リアルタイム処理
* UIとバックエンドの接続
* Windows起動スクリプト

推奨フロー:

```text
Codexが実装
→ Claude Codeがレビュー
→ Codexがレビュー指摘を反映
```

または、

```text
Gemini CLIが初稿
→ Codexが修正・統合
→ Claude Codeがレビュー
→ Codexがレビュー指摘を反映
```

---

### Q5. テストを必ず追加する

最低限、以下はテスト対象にする。

#### key_estimator

* C majorらしい音列でC majorが上位に来る
* A minorらしい音列でA minorが上位に来る
* 変な入力でもクラッシュしない
* 空入力でもクラッシュしない

#### chord_estimator

* C/E/Gが多いとC候補が上位に来る
* A/C/Eが多いとAm候補が上位に来る
* 単音だけでも候補を返す
* 空入力でも安全に返す

#### progression_smoother

* コードが毎拍ガチャガチャ変わらない
* 直前コードから極端に遠い候補を抑制できる
* 小節頭または拍頭でだけコード更新できる

#### accompaniment_engine

* 指定コードからベース音を生成できる
* 指定コードからパッド音を生成できる
* テンポ変更でタイミングが破綻しない
* 未知コードでクラッシュしない

---

### Q6. 実機確認と自動テストを分ける

MIDIや音声は実機依存があるため、全てを自動テストにしようとしない。

自動テスト対象:

* 音楽理論ロジック
* 推定ロジック
* データ変換
* パターン生成
* スムージング

手動テスト対象:

* MIDIキーボード入力
* FluidSynth再生
* Windowsオーディオ遅延
* UIの体感
* ライブデモ安定性

---

### Q7. 品質ゲートを通らない変更は採用しない

mainに入れる前に以下を満たす。

```text
- 起動できる
- 既存テストが通る
- 新規テストがある
- READMEまたはdocsが必要に応じて更新されている
- 変更範囲が説明されている
- 未解決の懸念が明記されている
```

## Non-Goals

現段階では以下を目指さない。

* 複数AI CLIを統合する巨大な開発カーネル
* 完全自律開発エージェント基盤
* 深層学習による本格作曲
* 音声/鼻歌入力
* DAWプラグイン化
* 商用品質の音源
* 全ジャンル対応
* 全キー・全スケールの完全対応
* 完璧なコード進行推定
* 大規模GUIアプリ化

まずは展示MVPを優先する。

## Milestone 1: 推定UI

目的:

> MIDI入力を受け取り、推定結果を画面に表示する。

実装対象:

* MIDI入力
* メロディバッファ
* キー推定
* コード候補推定
* 簡易UI
* ログ表示
* テスト

この段階では伴奏音が鳴らなくてもよい。

成功条件:

```text
- MIDIキーボード入力を受け取れる
- 入力ノートが画面またはログに表示される
- 直近ノート履歴が見える
- キー候補が表示される
- コード候補が表示される
- 空入力や異常入力で落ちない
```

---

## Milestone 2: 簡易伴奏

目的:

> 推定コードに基づいて、簡単な伴奏を鳴らす。

実装対象:

* ドラムループ
* ベースルート
* パッドコード
* 小節頭でのコード切替
* 確信度に応じた伴奏密度変更

成功条件:

```text
- 推定コードに応じて伴奏が変わる
- コードが細かく揺れすぎない
- 伴奏が破綻しない
- Windows PCで再生できる
```

---

## Milestone 3: 展示デモ

目的:

> Maker Faire / ハッカソンで見せられるライブデモにする。

実装対象:

* ジャンル切替
* デモ用プリセット
* 起動スクリプト
* ライブデモ台本
* README整備
* トラブルシュート

成功条件:

```text
- 1分程度のライブデモが安定してできる
- 技術説明なしでもコンセプトが伝わる
- UIでAIが推定している様子が見える
- 同じメロディでジャンル差が出る
```
