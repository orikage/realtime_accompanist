# Feasibility

## 結論

MVPは実現可能性が高い。
ただし、売り文句とスコープを正しく切る必要がある。

避けるべき表現:

> メロディだけから正しいコードを完全推定します。

採用すべき表現:

> メロディからキー・コード候補を推定し、確信度に応じてリアルタイム伴奏します。

## 機能別評価

| 機能 | 難易度 | 実現性 | コメント |
|---|---:|---:|---|
| MIDI入力 | 低 | 高 | Python mido/rtmidi等で可能 |
| キー推定 | 低〜中 | 高 | ルールベースで十分 |
| コード候補推定 | 中 | 高 | 一意推定ではなく候補提示なら可能 |
| 自動伴奏 | 中 | 高 | パターン駆動で可能 |
| 低遅延再生 | 中 | 中〜高 | 音源とバッファ設定次第 |
| Web UI | 低〜中 | 高 | FastAPI/WebSocket等で実装可能 |
| ジャンル切替 | 中 | 高 | パターン差し替えで可能 |
| 観客参加UI | 低 | 高 | 展示向き |
| 音声入力 | 高 | 低〜中 | 初期MVPでは避ける |
| 深層学習作曲 | 高 | 中 | 後回し推奨 |

## 最大リスク

### 1. コード推定の曖昧さ

単音メロディだけではコードは決まらない。
対策:

- 候補として扱う
- 確信度を表示する
- 確信度が低い時は伴奏を薄くする
- 観客が候補を選べるようにする
- コード進行補正を入れる

### 2. 伴奏がガチャガチャ変わる

推定値をそのまま音にすると、コードが毎拍ブレて不自然になる。
対策:

- 小節頭または拍頭でのみコード切替
- 直前コードから遠い候補にペナルティ
- 一定時間ホールド
- ヒステリシスを入れる
- ベースの跳躍を抑える

### 3. 音がショボい

展示ではかなり重要。
対策:

- Lo-fiを初期スタイルにする
- 良いSoundFontを使う
- ドラムはサンプル再生にする
- リバーブ/コンプを軽くかける
- ベース/パッド音色を固定して作り込む

### 4. レイテンシ

リアルタイム楽器として気持ちよさを損なう。
対策:

- MIDI入力に限定
- 推定処理は軽量ルールベース
- 音源バッファを小さくする
- UI更新と音声再生を分離
- コード変更は拍頭/小節頭に予約

## 推奨実装方針

最初はPythonでよい。

候補:

- MIDI input: mido + python-rtmidi
- Scheduler: custom clock, asyncio, or simple tick loop
- Sound: FluidSynth, pygame.midi, or external DAW MIDI out
- UI: FastAPI + WebSocket + React/Vite or simple HTML

最初のマイルストーンでは音を鳴らす前に、以下を完成させる。

1. MIDI入力を取る
2. 入力ノート履歴を表示
3. キー候補を表示
4. コード候補を表示
5. 1小節ごとにコードを決める

ここまでできれば、伴奏生成はパターン追加で進められる。
